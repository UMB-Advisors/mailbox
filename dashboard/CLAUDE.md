## Project

**MailBox One Dashboard**

Standalone Next.js 14 dashboard for the MailBox One T2 appliance. Exposes a human-in-the-loop approval queue for LLM-generated email drafts; on approve, triggers a real Gmail send via an n8n webhook.

This closes Phase 1 deliverable #6 (dashboard approval queue) and ships workflow #3 (send pipeline) of the MailBox One product roadmap.

**Core Value:** The operator can review, edit, approve, or reject LLM-drafted email replies on their phone in under 30 seconds, and approval results in a real Gmail reply going out. Without the dashboard, drafts sit in `mailbox.drafts` with no path to send.

## Technology Stack

Defer to root `../CLAUDE.md` for the appliance-wide stack. Dashboard-specific layers:

- **Next.js 14.2.x** (App Router) — internal API routes under `app/api/**/route.ts`; pages under `app/**/page.tsx`
- **Kysely 0.28.x** + **`pg` 8.x** — typed SQL query builder over Postgres `mailbox` schema. `getKysely()` returns `Kysely<DB>` where `DB` is generated by `kysely-codegen` into `lib/db/schema.ts`. Pool client in `lib/db.ts`; `getPool()` exposes the underlying `pg.Pool` for the migration runner, `sql.raw` escape hatch, and test helpers. Plain `.sql` migrations under `migrations/NNN-*.sql` run via `migrations/runner.ts` (custom tsx script, no drizzle-kit, no prisma-migrate). Per 2026-05-01 Dashboard ORM ADR (Kysely chosen over Prisma/Drizzle on Jetson hardware grounds — see root `../CLAUDE.md` Active decision records and `.planning/STATE.md`)
- **`kysely-codegen` 0.20.x** (devDep) — `npm run db:codegen` regenerates `lib/db/schema.ts` from `test/fixtures/schema.sql` via a temp postgres:17-alpine container. CI verifies via `npm run db:codegen:verify`
- **Tailwind CSS v3.4.x** — utility classes; mobile-responsive approval queue
- **zod 4.4.x** — runtime validation (STAQPRO-138 shipped 2026-05-01). `z.object({...})` schemas in `lib/schemas/{common,drafts,internal}.ts`, parsed by shared `lib/middleware/validate.ts` (`parseJson`, `parseParams`, `parseQuery` helpers). Structured 400 on validation failure

The dashboard runs as the `mailbox-dashboard` service in the appliance Docker Compose stack. It is reachable behind Caddy at `https://mailbox.heronlabsinc.com/dashboard/queue` (basic_auth gated per STAQPRO-131) and over the LAN/Tailnet on port `3001`.

## Conventions

> Source of truth lives in root `../CLAUDE.md` Conventions section. This file cross-references the dashboard-relevant subset; do not duplicate.

- **Route handler pattern** — App Router contract: export named `GET`/`POST`/`PATCH`/`DELETE` from `app/api/**/route.ts` accepting `(request: Request, { params })`. Internal routes under `/api/internal/*` are not Caddy basic_auth gated — they are called from n8n inside the docker network. Treat them as trust-boundary inputs anyway (zod-validate per STAQPRO-138).
- **Status state machine** — `mailbox.drafts.status` lifecycle (live CHECK): `pending` → `awaiting_cloud` (cloud call in flight) → (`approved` | `rejected` | `edited`) → (`sent` | `failed`). SoT = `DRAFT_STATUSES` const tuple in `lib/types.ts` (STAQPRO-137 shipped); asserted against the live Postgres CHECK constraint in `migrations/003-evolve-drafts-to-queue-shape-v1-2026-04-27.sql` by `test/schema-invariants.test.ts`.
- **Draft source values** — `drafts.draft_source` = `local` | `cloud` (the route taken). The model used is recorded separately in `drafts.model`. Constraint also permits legacy `local_qwen3` / `cloud_haiku` but the live drafting path writes `local` / `cloud`.
- **SQL convention** — Typed query builder via `getKysely()` from `lib/db.ts`. Three surface patterns: (a) named query helpers in `lib/queries*.ts` (preferred — keeps query construction out of route handlers, exposes a typed surface), (b) inline `db.selectFrom('drafts')...` calls inside a route handler when the query is one-off, and (c) `sql\`...\`` template tag escape hatch when raw SQL reads cleaner (e.g., a `CASE WHEN` expression in `.set()`). Promote to `lib/queries*.ts` when 2+ routes need the same query. Always use parameter binding via `eb.ref()` / `${value}` in sql templates — never string-concatenate user input.
- **Type SoT boundary** — `lib/types.ts` holds *semantic* types (const tuples + string-literal unions for status/source/category/stage) and *curated views* (Draft, InboxMessage, etc — narrower than the full DB row). `lib/db/schema.ts` is the kysely-codegen output and holds full DB row shapes; re-exported as `DraftRow`, `InboxMessageRow`, etc. from `lib/types.ts` when callers need the full surface.
- **Validation** — request body / query / params validation goes through zod schemas in `dashboard/lib/schemas/` via the shared `dashboard/lib/middleware/validate.ts` middleware. No inline `typeof x !== 'string'` checks in handlers.
- **Internal route auth** — `/api/internal/*` callers should send the n8n internal HMAC header (when wired). Until then, treat exposure as the threat model and zod-validate aggressively.
- **Migration comments** — every `.sql` migration opens with a 2-3 line block comment per the migration 007 standard: what, why (link issue or DR), reversal note. The comment is the only documentation — there's no separate ADR per migration.
- **`rag_context_refs` field semantics (STAQPRO-191)** — `mailbox.drafts.rag_context_refs` is a `jsonb` array of Qdrant point UUIDs (deterministic via `pointIdFromMessageId(message_id)`) that were retrieved into the drafting prompt at draft-time. Empty `[]` is the no-retrieval baseline (no hits, infra failure, or cloud_gated). The companion `rag_retrieval_reason` column (TEXT, default `'none'`) discriminates the empty case — values: `'ok' | 'cloud_gated' | 'embed_unavailable' | 'no_hits' | 'qdrant_unavailable' | 'none'`. Both columns mirror onto `mailbox.sent_history` via the migration 013 archival trigger so STAQPRO-192's eval delta can join them post-send. Writeback happens unconditionally in `app/api/internal/draft-prompt/route.ts` — never gated on `refs.length > 0` so the audit chain captures retrieval attempts even when they return empty. Replay: `curl -X POST $QDRANT_URL/collections/email_messages/points -H 'content-type: application/json' -d '{"ids":[...]}'` resolves the UUIDs back to source emails.

## Architecture

The dashboard plays two roles in the appliance:

1. **Operator UI** — approval queue at `/dashboard/queue`. Renders pending drafts from `mailbox.drafts` (status `pending_approval`); operator can approve/reject/edit. On approve, the dashboard fires the `MailBOX-Send` sub-workflow via the n8n webhook URL in `N8N_WEBHOOK_URL`.
2. **Internal API for n8n** — n8n's classify/draft sub-workflows call back into the dashboard for prompt assembly, output normalization, and live-gate checks. These routes live under `/api/internal/*` and are reached over the docker network.

### Routes (App Router)

CRUD (operator-facing, basic_auth gated by Caddy):
- `app/api/drafts/route.ts` — `GET` list of pending drafts
- `app/api/drafts/[id]/route.ts` — `GET` single draft
- `app/api/drafts/[id]/approve/route.ts` — `POST` approve + fire send webhook
- `app/api/drafts/[id]/reject/route.ts` — `POST` reject
- `app/api/drafts/[id]/edit/route.ts` — `PATCH` edit body before approve
- `app/api/drafts/[id]/retry/route.ts` — `POST` re-run draft generation

Internal (n8n-facing, docker network only):
- `app/api/internal/classification-prompt/route.ts` — assemble classify prompt for the Qwen3 call
- `app/api/internal/classification-normalize/route.ts` — normalize Qwen3 classify output → enum + confidence
- `app/api/internal/draft-prompt/route.ts` — assemble drafting prompt (route-aware: local vs cloud persona blocks)
- `app/api/internal/draft-finalize/route.ts` — accept LLM draft output, persist to `mailbox.drafts`, set `status = pending_approval`
- `app/api/onboarding/live-gate/route.ts` — gate classify/draft execution behind onboarding completion. Bypass via `MAILBOX_LIVE_GATE_BYPASS=1` for dogfood

### Data layer

- `lib/db.ts` — `pg.Pool` (lazy from `POSTGRES_URL`) wrapped by `getKysely(): Kysely<DB>`. `setTypeParser(1184/1114)` overrides keep TIMESTAMP/TIMESTAMPTZ as strings on both surfaces. `getPool()` is still exported for the migration runner, the `sql.raw` escape hatch, and test setup/teardown helpers.
- `lib/db/schema.ts` — kysely-codegen output. Full DB row shapes for the `mailbox` schema. Regenerate via `npm run db:codegen`. CI drift check: `npm run db:codegen:verify`.
- `lib/queries.ts` — drafts queue + dashboard CRUD helpers (kysely)
- `lib/queries-onboarding.ts` — onboarding state queries (kysely)
- `lib/queries-persona.ts` — persona config queries (kysely, includes the `customer_key` upsert with `excluded.*` references)
- `lib/transitions.ts` — `transitionToApprovedAndSend` (DRY approve/retry from STAQPRO-140, kysely)
- `lib/types.ts` — semantic SoT (const tuples + string unions) + curated views (Draft, InboxMessage, …) + `Selectable<…>` row re-exports (DraftRow, InboxMessageRow, …)

Migrations live as plain `.sql` files in `migrations/NNN-*.sql`. The runner is `migrations/runner.ts` (custom tsx script that tracks applied versions in `mailbox.migrations`). The separate `mailbox-migrate` compose service runs the same script: `docker compose --profile migrate run mailbox-migrate`. Kysely does not own migrations.

### Routing logic

`dashboard/lib/classification/prompt.ts:routeFor` decides local vs cloud per the rules in root `../CLAUDE.md` (Architecture > Routing rules). When this file is edited, also re-run the eval baselines (see `scripts/eval-output-2026-05-01-tuned.md` for the live behavior reference).

### Tests

Vitest runs 44 cases via `npm test` (STAQPRO-133 v1+v2+v3 shipped 2026-05-01):
- `lib/middleware/validate.test.ts` — 19 zod-validation cases
- `test/schema-invariants.test.ts` — 5 cases asserting the `lib/types.ts` const tuples match the live Postgres CHECK constraints
- `test/routes/internal.test.ts` — 4 cases covering `/api/internal/*` happy paths
- `test/routes/drafts.test.ts` — 13 cases covering CRUD routes
- `test/routes/pipeline-smoke.test.ts` — 3 cases for the classify → draft → finalize chain

DB-backed tests skip without `TEST_POSTGRES_URL`. CI bootstraps a postgres:17-alpine service from `test/fixtures/schema.sql` (the canonical snapshot — same path `npm run db:codegen` takes) per STAQPRO-134. Local: `ssh -L 5432:localhost:5432 jetson-tailscale -N` then `TEST_POSTGRES_URL=postgresql://mailbox:<pw>@localhost:5432/mailbox npm test`.

Infrastructure smoke remains: `scripts/smoke-test.sh` (GPU/Qdrant/Postgres health) and the in-container `smoke-draft.mjs`.
