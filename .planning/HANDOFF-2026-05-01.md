## Surprise gotchas from prior session work

These are things the new Claude session will trip over if not warned:

1. **Cloud LLM vendor is undecided.** STAQPRO-156's comment thread
   proposes routing the 02-07 cloud path through Ollama Cloud
   (`gpt-oss:120b`) instead of Anthropic Haiku 4.5. The 02-07 PLAN.md
   (commit `2eed824`) was written assuming Anthropic. Until the
   decision lands, treat both as live possibilities; do not edit
   02-07's D-41..D-45 unilaterally.

2. **n8n Postgres node Insert mode is comma-safe; Execute Query is not.**
   The original Execute Query mode parsed parameters by splitting
   queryReplacement on commas, which silently broke on email bodies
   containing commas. Use Insert mode with column mapping for any
   new Postgres writes from n8n.

3. **n8n expression mode and the = prefix.** When typing in an
   Expression-mode field in n8n's UI, do NOT type the leading `=`.
   The `=` is implicit; typing it produces values like `={{ ... }}`
   that work for string columns but break for numeric columns.

4. **Gmail node returns thread headers as top-level camelCase fields:**
   `$json.inReplyTo` and `$json.references` (NOT under `$json.headers`).
   First emails in a thread have these as undefined, which n8n correctly
   converts to NULL on insert.

5. **Schedule Trigger UI quirk.** Renaming the node does not change
   the trigger interval. The interval lives in the
   `rule.interval[].minutesInterval` field which the UI sometimes fails
   to persist. If you change the interval, verify by querying n8n's
   `workflow_entity` table directly and reading `nodes::jsonb`.

6. **Workflow JSON drift.** The committed `n8n/workflows/*.json` files
   reflect the live workflows as of recent exports. There is no
   automated import/export — operator edits the workflow in n8n's UI,
   then re-exports the JSON, then commits. Drift is detectable manually
   but not automated. STAQPRO-139 tracks the long-term sync process.

7. **Drizzle vs Prisma is contested.** STAQPRO-136 (open, High) wants
   to migrate to Prisma; the 2026-04-27 Next.js full-stack ADR locks
   in Drizzle. **Do not start any DB-layer refactor work until that
   ticket is resolved.** Comment thread on STAQPRO-136 has the full
   tradeoff analysis.

8. **Ollama and Qdrant are no longer LAN-reachable from the Jetson.**
   Per STAQPRO-130 (2026-05-01 ship), `docker-compose.yml` no longer
   publishes `:11434` or `:6333` on the host. CLAUDE.md still says
   they are LAN-only — that doc is stale; STAQPRO-117 will fix.
   For dev access: `docker compose exec -T ollama …` or use the
   future `docker-compose.dev.yml` (STAQPRO-155).

## Tools and conventions

- **Migration runner:** ephemeral Docker service `mailbox-migrate`,
  invoked via `docker compose --profile migrate run --rm mailbox-migrate`.
  Runs forward-only SQL files from `dashboard/migrations/` in lexical
  order. Tracks applied versions in `mailbox.migrations`.
- **Filename versioning:** per Dustin's preference, all migration and
  plan files include version-history suffixes
  (`-v1-YYYY-MM-DD.sql`, `-v2-YYYY-MM-DD.md`, etc.).
- **Commit messages:** Conventional Commits style with `feat(scope)`
  or `docs(scope)` scope. Body explains the why and references the
  decision IDs from the addendum where relevant. For Linear-tracked
  work, include the issue ID in the scope: `fix(infra/STAQPRO-130): …`.
- **Stub vs PLAN vs SUMMARY:** Stubs (suffix `-STUB.md`) capture shape
  + decisions; not directly executable. Full PLANs are executable
  (e.g. `02-07-...-PLAN.md`). SUMMARYs (`-SUMMARY-vN-YYYY-MM-DD.md`)
  document what shipped. v1 PLANs in the original `*-PLAN.md` form
  are historical artifacts — almost all carry SUPERSEDED frontmatter.
