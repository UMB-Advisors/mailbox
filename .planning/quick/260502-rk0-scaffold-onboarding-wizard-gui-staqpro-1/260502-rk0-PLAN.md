<wizard_to_db_stage_mapping>
The CHECK constraint on `mailbox.onboarding.stage` is fixed (migration 006) and
MUST NOT be modified in this scaffold. The 6 wizard UX steps map to existing
DB stages as follows:

| Wizard step path        | DB stage on entry      | Notes |
|-------------------------|------------------------|-------|
| /onboarding/welcome     | `pending_admin`        | Default seed stage. Entry point. |
| /onboarding/password    | `pending_admin`        | UX-only step inside `pending_admin`; no DB flip. Real flip awaits STAQPRO-131 password landing. |
| /onboarding/profile     | `pending_email`        | Wizard advances DB to `pending_email` on Next from password. |
| /onboarding/network-check | `pending_email`      | UX-only step; no DB flip. Real flip awaits Caddy-cert health check spec. |
| /onboarding/email-connect | `ingesting`          | Wizard advances DB to `ingesting` on Next from network-check. Real OAuth deferred. |
| /onboarding/complete    | `live`                 | Wizard advances DB to `live` on the explicit "Finish" action. Skips `pending_tuning` / `tuning_in_progress` for the scaffold (those need the tuning-loop spec). |

This mapping lives in `dashboard/lib/onboarding/wizard-stages.ts` as the single
source of truth: an ordered tuple of step records (slug, title, intent, dbStage,
allowsBack). Both the wizard pages and the advance route import from there so a
future stage rename is one file.

The advance route's allowed forward transitions are exactly the adjacent pairs
in this tuple PLUS the welcome -> password no-op (same DB stage); any other
incoming `(from, to)` pair is rejected as `invalid_transition`.
</wizard_to_db_stage_mapping>
